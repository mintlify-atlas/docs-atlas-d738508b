JustBenchThatLLM provides comprehensive health check endpoints and monitoring capabilities for all components. This guide covers how to monitor your deployment effectively.

## Health Check Endpoints

### Backend API

The backend exposes two health endpoints on port `8000`:

<CodeGroup>
```bash Liveness Check
curl http://backend:8000/health
```

```bash Readiness Check
curl http://backend:8000/ready
```
</CodeGroup>

**Configuration** (from values.yaml:180-196):
- **Liveness probe**: Checks if the backend is running
  - Initial delay: 30 seconds
  - Check interval: 10 seconds
  - Timeout: 5 seconds
  - Failure threshold: 3 attempts

- **Readiness probe**: Checks if the backend can accept requests
  - Initial delay: 10 seconds
  - Check interval: 5 seconds
  - Timeout: 3 seconds
  - Failure threshold: 3 attempts

### Worker

The worker exposes health endpoints on port `8006`:

<CodeGroup>
```bash Liveness Check
curl http://worker:8006/health
```

```bash Readiness Check
curl http://worker:8006/ready
```
</CodeGroup>

**Configuration** (from values.yaml:256-273):
- **Liveness probe**: Ensures worker process is alive
  - Initial delay: 30 seconds
  - Check interval: 10 seconds
  - Timeout: 5 seconds
  - Failure threshold: 3 attempts

- **Readiness probe**: Ensures worker can process benchmarks
  - Initial delay: 10 seconds
  - Check interval: 5 seconds
  - Timeout: 3 seconds
  - Failure threshold: 3 attempts

## Database Health Checks

### Redis

Monitor Redis connectivity and performance:

<CodeGroup>
```bash Docker Compose
# Redis health check runs automatically
docker-compose ps redis

# Manual check
docker exec redis redis-cli ping
# Expected: PONG
```

```bash Kubernetes
# Check Redis pod status
kubectl get pods -l app.kubernetes.io/component=redis

# Execute health check
kubectl exec -it <redis-pod> -- redis-cli ping
```
</CodeGroup>

**Health check configuration** (from values.yaml:59-64):
- Initial delay: 10 seconds
- Check interval: 5 seconds
- Timeout: 3 seconds
- Success threshold: 1
- Failure threshold: 5 attempts

### MongoDB

Monitor MongoDB connectivity and replication status:

<CodeGroup>
```bash Docker Compose
# MongoDB health check runs automatically
docker-compose ps mongo

# Manual check
docker exec mongodb mongosh --eval "db.adminCommand('ping')"
```

```bash Kubernetes
# Check MongoDB pod status
kubectl get pods -l app.kubernetes.io/component=mongo

# Execute health check
kubectl exec -it <mongo-pod> -- mongosh --eval "db.adminCommand('ping')"
```
</CodeGroup>

**Health check configuration** (from values.yaml:112-118):
- Initial delay: 30 seconds
- Check interval: 5 seconds
- Timeout: 3 seconds
- Success threshold: 1
- Failure threshold: 5 attempts

## Component Status Monitoring

### Docker Compose

Monitor all components with Docker Compose:

```bash
# Check status of all services
docker-compose ps

# View logs for specific component
docker-compose logs -f backend
docker-compose logs -f worker

# Check resource usage
docker stats
```

### Kubernetes

Monitor components in Kubernetes:

<CodeGroup>
```bash Pod Status
# Check all pods
kubectl get pods

# Check specific component
kubectl get pods -l app.kubernetes.io/component=backend
kubectl get pods -l app.kubernetes.io/component=worker
```

```bash Resource Usage
# View resource consumption
kubectl top pods

# Detailed pod information
kubectl describe pod <pod-name>
```

```bash Logs
# Stream logs
kubectl logs -f deployment/justbenchthatllm-backend
kubectl logs -f deployment/justbenchthatllm-worker

# View recent logs
kubectl logs --tail=100 deployment/justbenchthatllm-backend
```
</CodeGroup>

## Integration with Monitoring Tools

### Prometheus Metrics

If using Prometheus, monitor these key metrics:

```yaml Example ServiceMonitor
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: justbenchthatllm
spec:
  selector:
    matchLabels:
      app.kubernetes.io/name: justbenchthatllm
  endpoints:
  - port: http
    interval: 30s
    path: /metrics
```

### Log Aggregation

Configure log levels through environment variables (values.yaml:151, 224):

```yaml
backend:
  env:
    LOG_LEVEL: INFO  # Options: DEBUG, INFO, WARNING, ERROR

worker:
  env:
    LOG_LEVEL: INFO
```

<Warning>
Setting `LOG_LEVEL: DEBUG` significantly increases log volume and may impact performance. Only use DEBUG level for troubleshooting.
</Warning>
