### Latency Distribution

**Understanding percentiles** - Why averages aren't enough:

- **Minimum**: Best-case latency (usually cached or optimized requests)
- **P50 (Median)**: Typical user experience - 50% of requests are faster
- **P90**: Performance threshold - 90% of users get this or better
- **P95**: Quality target - catches most outliers
- **P99**: Tail latency - reveals worst-case scenarios affecting 1% of requests
- **Maximum**: Absolute worst case (often indicates timeouts or errors)

<Info>
P95 and P99 are critical for SLO definitions because they represent the experience of your slowest users, not just the average.
</Info>

#### Why Percentiles Matter

<Tabs>
  <Tab title="P50 (Median)">
    **The typical experience**

    - Half of your users see better latency, half see worse
    - More stable than average (not skewed by outliers)
    - Good for understanding baseline performance
    - Use for: Comparing typical performance between configurations
  </Tab>

  <Tab title="P90">
    **The quality bar**

    - 90% of requests complete within this time
    - Catches most performance issues while ignoring rare outliers
    - Common SLO target (e.g., "P90 latency < 500ms")
    - Use for: Setting realistic performance targets
  </Tab>

  <Tab title="P95 & P99">
    **Tail latency - the user experience killers**

    - P95: 1 in 20 requests is this slow or slower
    - P99: 1 in 100 requests is this slow or slower
    - Critical for high-volume applications
    - Often reveals capacity or resource contention issues
    - Use for: Identifying performance bottlenecks and outliers

    <Note>
    A system with P50=100ms but P99=5000ms means 1% of users wait 50x longer than typical. That's a terrible experience for high-traffic applications.
    </Note>
  </Tab>
</Tabs>

### Time To First Token (TTFT)

**Why TTFT matters for streaming applications:**

TTFT measures the delay before the first token arrives - critical for perceived responsiveness:

- **Low TTFT (< 200ms)**: Feels instant, excellent user experience
- **Medium TTFT (200-500ms)**: Noticeable but acceptable delay
- **High TTFT (> 500ms)**: Perceivable lag, may impact user experience

<Tip>
For streaming chat applications, TTFT is often more important than total latency. Users perceive fast TTFT as responsive even if full generation takes longer.
</Tip>

**TTFT Analysis:**
- Compare TTFT across concurrency levels to identify queueing delays
- Track TTFT percentiles to catch cold start issues
- Monitor TTFT degradation during stepped load tests

### Throughput Metrics

**Understanding capacity:**

- **Requests per second (RPS)**: How many requests completed per second
- **Tokens per second**: Total token generation rate
- **Tokens per second per request**: Average generation speed

**Throughput insights:**

<img src="/img/result_03.png" alt="Throughput Analysis" />

- **Constant throughput** across concurrency levels = linear scaling
- **Decreasing throughput** at high concurrency = resource saturation
- **Throughput vs latency trade-off**: Higher concurrency may increase throughput but degrade latency

### Success Rates

**Reliability under load:**

- **Success rate**: Percentage of requests that completed without errors
- **Error types**: Timeouts, 5xx errors, connection failures
- **Degradation patterns**: When errors begin to occur (capacity limit)

<Note>
- 100% success rate at low concurrency is expected
- Success rate drops during stepped load indicate maximum capacity
- Sudden drops often reveal configuration limits (connection pools, memory, etc.)
</Note>

### Stepped Load Analysis

Stepped load benchmarks automatically generate progressive load curves:

<img src="/img/result_04.png" alt="Stepped Load Analysis" />

**What stepped load reveals:**
- **Linear scaling region**: Where adding concurrency increases throughput without degrading latency
- **Degradation point**: When latency begins to increase significantly
- **Maximum capacity**: Where errors start occurring or latency becomes unacceptable
- **Resource saturation**: Identify which resource (CPU, GPU, memory, bandwidth) becomes the bottleneck

<Tip>
Use stepped load testing to find your optimal operating point - the highest concurrency where latency and success rate remain acceptable.
</Tip>

## Request-Level Analysis

### Timeline View - Deep Dive

<img src="/img/timeline.jpg" alt="Timeline View" />

The interactive timeline view provides request-by-request detail:

**What you can see:**
- Individual request latencies plotted over time
- TTFT for each request
- Success vs error status
- Request timing and concurrency patterns
- Outliers and anomalies

**Use timeline view to:**
- **Identify patterns**: Regular slowdowns, periodic spikes, warm-up effects
- **Spot anomalies**: Outlier requests that skew percentiles
- **Debug issues**: Correlate errors with timing or concurrency
- **Validate results**: Ensure benchmark ran as expected
- **Understand behavior**: See how requests overlap and compete

### Drilling Into Outliers

When you spot anomalous requests in the timeline:

1. **Click on outlier requests** to see detailed information
2. **Review request parameters** to understand what was different
3. **Check timing** - was it during high concurrency or another benchmark event?
4. **Compare with endpoint logs** to identify root cause
5. **Determine if outliers are acceptable** or indicate issues to fix

<Note>
Outliers often reveal:
- Cold start penalties
- Garbage collection pauses
- Resource contention
- Network issues
- Queue saturation
</Note>

## Best Practices

### Analysis Workflow

1. **Start with high-level metrics**: Review radar charts and summary statistics
2. **Drill into anomalies**: Use overlay graphs to understand divergences
3. **Examine request-level data**: Timeline view reveals patterns and outliers
4. **Compare with expectations**: Validate results match anticipated behavior
5. **Document insights**: Record findings and decisions for future reference

### Avoiding Common Pitfalls

<Note>
**Don't:**
- Rely solely on average latency (use percentiles)
- Ignore tail latency (P95/P99 matter for user experience)
- Compare benchmarks with different warmup states
- Draw conclusions from single runs (validate with multiple benchmarks)
- Skip timeline analysis (request-level patterns reveal important insights)

**Do:**
- Focus on P95/P99 for SLO definitions
- Run multiple benchmarks to validate consistency
- Consider TTFT for streaming applications
- Use stepped load to understand scalability
- Export data for long-term trend tracking
</Note>
